# 拾时 App 产品需求文档（PRD）

> 产品名称：拾时  
> 文档版本：V0.2  
> 文档状态：功能规划稿  
> 产品阶段：正式开发前需求定义  
> 产品形态：手机 App  
> 核心定位：以时间为线索，随手记录生活、追加后续，并在未来重新遇见过去。

## 1. 产品概述

### 1.1 产品背景

用户会在一天中的不同时间突然想到或遇到值得记录的事情。传统日记通常要求用户在一天结束时集中整理，容易遗忘当时的细节，也会给人“必须认真写一篇文章”的压力。

拾时提供类似动态信息流的轻量记录体验，但不包含社交和公开发布能力。用户可以随时写下一条生活记录，系统自动保存当时的时间；以后可以通过时间轴、日历和搜索回看，也可以在原记录下添加带有独立时间的“后续”。

### 1.2 产品目标

- 让用户从打开 App 到完成一条文字记录，尽可能在 10 秒内完成。
- 自动、准确地保存记录时刻，降低输入负担。
- 用熟悉的卡片信息流呈现记录，降低学习成本。
- 数据默认保存在本机，由用户掌控，不依赖公开账号或社交关系。
- 随着记录增多，仍能通过日期和关键词快速找到过去的内容。
- 允许用户在旧记录下追加带时间的“后续”，补充事情的发展和新的想法，而不必改写原始记录。
- 产品从第一天起为 iOS、Android、多设备同步和未来上架预留清晰的演进路径。
- 通过“拾起一刻”、那年今日和关联记录，让已经写下的内容在未来继续产生价值。

### 1.3 非目标

MVP 不包含：

- 公开发布、分享动态或生成公开链接。
- 好友、关注、粉丝、点赞、他人评论、私信。
- 排行榜、热门内容和推荐算法。
- 社区运营、内容审核和广告。
- 多人共同编辑日记。
- AI 总结、AI 陪伴等智能功能。

### 1.4 品牌与产品语言

#### 产品名称

- 中文名：**拾时**
- 名称含义：既表示随手拾起正在发生的时刻，也表示未来重新拾起过去的记忆。
- 暂定 Slogan：**把日子慢慢收好。**
- 备用 Slogan：**把生活留在它发生的时刻。**

#### 核心产品用语

| 场景 | 界面文案 | 含义 |
| --- | --- | --- |
| 品牌名称 | 拾时 | App 名称 |
| 首页内容区 | 我的日迹 | 用户生活记录的集合，可作为首页标题或副标题 |
| 主导航 | 时间轴 | 保留直观名称，帮助用户理解功能 |
| 新建记录 | 记录此刻 | 发布按钮与编辑器标题 |
| 补充进展 | 添加后续 | 在旧记录下追加新的内容 |
| 回忆入口 | 拾起一刻 | 随机或按条件重新展示过去的记录 |

“我的日迹”不是“时间轴”的替代名称。底部导航和功能说明统一使用“时间轴”，首页可呈现“拾时 / 我的日迹”的品牌层级。

产品不在首页和日常操作中反复强调私密属性。数据可见范围只在首次引导、同步说明、安全设置和隐私政策等必要位置准确说明。

## 2. 目标用户与核心场景

### 2.1 目标用户

- 希望记录日常，但难以坚持每天写长日记的人。
- 喜欢动态信息流的轻松记录方式，但不希望加入社交关系的人。
- 重视数据隐私，希望自己保存和管理记录的人。

### 2.2 核心场景

#### 场景 A：即时记录

用户刚经历一件小事，打开 App，输入一句话并保存。系统自动记录当前时间，不要求填写标题、分类或其他信息。

#### 场景 B：补记过去发生的事情

用户晚上想起上午发生的事情。App 仍自动保存当前的“记录时间”，用户可以另外修改“发生时间”为上午。

#### 场景 C：连续回看

用户沿时间轴向下滚动，查看过去的文字和图片，感受生活在时间上的连续变化。

#### 场景 D：寻找特定记忆

用户通过某一天、时间范围或正文关键词，找到过去的一条记录。

#### 场景 E：补充事情的后续

用户重新看到一条过去的记录，在详情页点击“添加后续”，写下事情后来的发展。原始记录保持不变，后续按照时间连续排列。

#### 场景 F：拾起过去的一刻

用户进入“拾起一刻”，随机看到过去的一条记录，或者查看那年今日、同一地点和同一标签下的内容。

## 3. 产品原则

1. **默认只需输入正文。** 标题、标签、心情等信息不能成为发布前的必填项。
2. **时间可信。** 自动保存记录时间；修改发生时间不能覆盖原始记录时间。
3. **默认安全。** 产品中不存在公开可见范围；数据保存、上传和同步状态必须准确、可理解。
4. **离线可用。** 没有网络时仍可新增、编辑、查看和搜索记录。
5. **数据可带走。** 用户可以备份和导出，不被单一 App 锁定。
6. **克制设计。** 页面服务于记录和回看，不用连续签到、红点或任务体系制造压力。

## 4. 信息架构

MVP 采用底部导航，包含三个主要入口：

| 入口 | 主要用途 |
| --- | --- |
| 时间轴 | 查看全部记录、搜索、进入详情 |
| 记录此刻 | 快速创建新记录；使用底部中央突出按钮 |
| 日历 | 按日期查看记录 |

设置入口放在时间轴页面右上角，不单独占用底部导航。

主要页面：

1. 启动与应用锁页面
2. 时间轴首页
3. “记录此刻”新建/编辑页
4. 记录详情页
5. “添加后续”区域（与详情页合并）
6. 日历页与单日记录列表
7. 搜索结果页
8. 设置页
9. 数据备份与恢复页
10. 登录与同步状态页（同步版本启用）
11. “拾起一刻”回忆页（回忆版本启用）
12. 收藏与故事页（增强版本启用）

## 5. 核心功能需求

优先级说明：P0 为首版必须实现，P1 为长期使用保障，P2 为回忆和组织能力，P3 为快捷与多媒体能力，P4 为账号与多设备能力。

### 5.1 新建记录（P0）

#### 功能描述

用户通过“记录此刻”创建一条生活记录。

#### 字段

| 字段 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| 正文 | 是 | 空 | MVP 支持纯文本和换行 |
| 记录时间 | 系统生成 | 点击保存时的本机时间 | 不允许用户直接修改 |
| 发生时间 | 是 | 打开新建页时的本机时间 | 用户可按需修改 |
| 图片 | 否 | 无 | 建议作为 P1 |
| 标签 | 否 | 无 | 建议作为 P1 |

#### 交互要求

- 点击记录按钮后直接聚焦正文输入框并弹出键盘。
- 页面显示当前发生时间，例如“今天 14:36”，点击后可修改。
- 保存按钮在正文为空时不可用。
- 点击保存后立即返回时间轴，并在顶部显示新记录。
- 快速连续点击保存不得产生重复记录。
- 退出未保存页面时，如果已有输入，弹窗询问是否放弃。
- 正文长度首版建议限制为 10,000 字，接近上限时提示。

#### 验收标准

- 无网络时可以正常创建记录。
- 保存后的记录时间精确到分钟，数据库内部精确到毫秒。
- 关闭并重新打开 App 后记录仍然存在。
- 跨日、跨月和跨年创建的记录排序正确。

### 5.2 时间轴（P0）

#### 功能描述

按发生时间倒序展示所有记录，视觉形式采用简洁的生活动态信息流。

#### 展示规则

- 记录按发生时间倒序排列；发生时间相同时按记录时间倒序排列。
- 每条卡片展示发生时间和正文摘要/全文。
- 当发生时间与记录时间差异明显时，在详情中展示“记录于某时”。
- 按日期自然分组，例如“今天”“昨天”“7 月 18 日”。
- 记录编辑后不改变其在时间轴中的位置，除非用户修改发生时间。
- 列表需要支持长数据量的虚拟化或分页加载。

#### 空状态

首次使用时显示：

> 从此刻开始，把日子慢慢收好。

并提供明显的“记录此刻”按钮。

#### 验收标准

- 新记录保存后无需重启即可显示在正确位置。
- 1,000 条纯文字记录下，首页滚动仍保持流畅。
- 正文换行、Emoji 和中英文混排正常显示。

### 5.3 记录详情、编辑和删除（P0）

- 点击时间轴卡片进入详情页。
- 详情展示完整正文、发生时间、记录时间和最后编辑时间。
- 用户可以修改正文和发生时间。
- 每次编辑更新“最后编辑时间”，但保留原始记录时间。
- 删除前需要二次确认。
- MVP 可先直接删除；更推荐在 P1 增加保留 30 天的回收站。

### 5.4 添加后续（P0）

#### 功能定位

“后续”是一条原始记录下的延伸时间线。用户日后翻到某条记录时，可以补充后来发生的事情、现在的想法或结果。

示例：

```text
7 月 19 日 14:30
今天第一次开始设计自己的日记 App。

后续 · 7 月 26 日 22:10
已经把时间轴和本地数据库做出来了。

后续 · 8 月 12 日 09:20
连续用了两周，发现搜索比标签更常用。
```

#### 功能要求

- 用户可以在记录详情页点击“添加后续”。
- 后续默认按创建时间正序排列，形成连续的事情发展过程。
- 每条后续自动保存创建时间，并记录最后编辑时间。
- 后续支持编辑和删除；删除后续不影响原记录。
- 没有后续时不在时间轴卡片中占用空间。
- 时间轴卡片默认展示最近一条后续；存在多条时同时显示“共 N 条后续”，点击进入详情查看全部。
- 新增后续不会改变原记录的 `occurredAt`、`createdAt` 或时间轴位置。
- 搜索默认同时检索正文和后续，并在结果中说明命中位置。
- P1 支持后续图片；MVP 后续仅支持文字。
- 可将一条后续提升为新的时间轴记录，同时保留与原记录的关联（P2）。

#### 验收标准

- 离线状态下可以新增、编辑和删除后续。
- 重新启动 App 后，后续内容、顺序和时间保持正确。
- 一条记录存在大量后续时，详情页仍可流畅加载。
- 删除原记录时，其后续应在同一数据库事务中一并删除或进入回收站。

### 5.5 日历回看（P0）

- 月历中有记录的日期显示圆点；可选显示当天记录数量。
- 点击日期后展示当天全部记录。
- 默认打开当前月份和今天。
- 支持切换上一个月/下一个月以及快速返回今天。
- 日期归属以用户当前时区下的发生时间为准。

### 5.6 搜索（P0）

- 支持按正文和后续关键词搜索。
- 支持中文、英文、数字和 Emoji 原文匹配。
- 结果按发生时间倒序展示，并高亮匹配词。
- 无结果时提供清晰空状态。
- P1 增加日期范围和标签组合筛选。

### 5.7 微博式记录编辑器（P0 基础版，P1 完整版）

编辑器采用类似微博发布动态的低门槛形式，而不是传统日记的“标题 + 正文”表单。

#### P0 基础版

- 打开后直接进入无标题正文输入区。
- 支持换行、Emoji、粘贴文本和系统输入法。
- 始终显示发生时间入口和保存按钮。
- 自动保存本地草稿，意外退出后可以恢复。
- 新建记录和编辑旧记录复用同一套编辑器，但页面标题和保存行为明确区分。
- 纯文字也可以直接保存，不强迫添加图片、标签或心情。

#### P1 完整版

- 支持最多 9 张图片、拖动排序和删除预览。
- 支持标签、心情和地点等可选信息。
- 支持图片与文字混合输入；是否进一步实现真正的富文本/图文穿插，在验证需求后决定。
- 提供字数提示、草稿箱和最近使用的标签。
- 暂不支持公开可见范围、@其他用户、话题广场和转发。

#### 编辑器边界

首版不建议实现字体、字号、颜色、对齐方式等复杂富文本格式。生活碎片的重点是快速输入；复杂富文本会显著增加跨平台渲染、搜索、导出和同步成本。

### 5.8 隐私与应用锁（P1，正式长期使用前建议完成）

- 支持系统生物识别或设备密码解锁。
- App 进入后台后，在系统任务切换界面中隐藏内容。
- 用户可设置“立即锁定”或离开一定时间后锁定。
- 本地数据库中的敏感内容应考虑加密；图片也需要纳入保护范围。
- 日志、崩溃报告和分析工具不得上传日记正文。

注意：MVP 若暂未实现数据库加密，产品页面不能宣称“端到端加密”或“加密存储”，只能准确描述为“数据保存在本机”。

### 5.9 备份、恢复与导出（P1）

- 用户可以主动导出完整备份文件。
- 支持从备份恢复，恢复前明确说明覆盖或合并策略。
- 支持导出通用 JSON，避免数据被 App 锁定。
- 后续可增加 Markdown 和 PDF 导出。
- 备份文件应包含格式版本号，以支持未来数据库升级。
- 若备份包含隐私内容，应支持设置备份密码。

### 5.10 图片（P1）

- 每条记录可添加多张图片，首版建议最多 9 张。
- 支持拍照和从相册选择。
- 图片按 App 私有目录管理，不仅保存系统相册路径。
- 删除记录时同步处理其专属图片。
- 备份时包含图片文件。

### 5.11 账号与多设备同步（P4）

#### 产品策略

- 第一版无需登录即可使用，避免账号系统阻碍记录。
- 开启同步时再引导用户创建账号或使用系统账号登录。
- 本地 SQLite 始终是主要读写来源；网络恢复后在后台增量同步。
- 未开启同步的用户，数据不应被默认上传。
- 同一账号支持 iOS 和 Android 设备使用。
- Web 端不是首发目标，但数据模型和 API 不应阻止未来增加只读或完整 Web 客户端。

#### 同步要求

- 支持离线新增、编辑、删除记录与后续。
- 每个实体使用客户端生成的 UUID，避免多设备离线创建时冲突。
- 保存版本号、更新时间、删除标记和设备标识等同步元数据。
- 删除采用墓碑记录同步，避免旧设备把已删除内容重新上传。
- 同一内容在两台设备同时编辑时，MVP 同步版可以使用“最后写入优先”，但必须保留冲突副本或历史版本，不能静默丢失正文。
- 图片先上传对象存储，确认成功后再提交附件元数据。
- 设置页展示最近同步时间、当前状态和可重试的失败项。
- 退出账号前明确提示未同步数据，并允许先导出备份。

#### 隐私要求

- 服务端数据库权限必须确保用户只能访问自己的数据。
- 生产环境传输必须使用 TLS。
- 若产品对外宣称服务端无法读取日记内容，则必须实现经过安全评审的客户端加密/端到端加密；不能仅依赖数据库权限或磁盘加密作此宣称。
- 端到端加密会影响搜索、密码找回、Web 端和多设备密钥恢复，作为正式上线前的独立技术设计项目处理。

## 6. 完整功能规划

### 6.1 P0：首个可用版本

- “记录此刻”：纯文字、自动时间、修改发生时间。
- 时间轴：按发生时间倒序、按日分组、查看详情。
- 编辑与删除记录。
- “添加后续”：文字后续、独立时间、编辑与删除。
- 日历回看。
- 搜索正文与后续。
- 自动保存和恢复当前编辑草稿。
- SQLite 本地持久化与数据库迁移。

首版完成标准不是功能数量，而是能够安装到自己的真实设备并连续使用。

### 6.2 P1：丰富记录与数据保障

- 每条记录最多 9 张图片，支持拖动排序。
- 后续支持图片。
- 标签、心情和地点；地点仅在用户主动选择时获取。
- 长文字折叠，详情页展示全文。
- 收藏记录。
- 保留 30 天的回收站。
- 编辑历史和版本恢复。
- 草稿箱。
- 应用锁和后台内容遮罩。
- 完整备份、恢复，以及 JSON/Markdown 导出。
- 深色模式、系统字体缩放和无障碍完善。

### 6.3 P2：拾起一刻与内容组织

“拾起一刻”是拾时的核心回忆入口，不替代时间轴。

- 随机展示一条过去的记录。
- 那年今日、一个月前的今天和一年前的本周。
- 按标签、地点或心情拾起过去的记录。
- 用户可以选择“不再推荐”某一条记录。
- 默认不强制弹窗，也不通过红点制造查看压力。
- 一天、一周、一个月的回顾。
- 年度记录热力图；用于呈现生活分布，不设计连续签到奖励。

#### 关联记录与故事

- 用户可以关联两条具有上下文关系的记录。
- 用户可以建立“故事”，将多条记录按发生时间组织在一起。
- 故事示例：开发第一个 App、一次旅行、养一盆植物。
- 后续可以提升为新记录，并保留与原记录的关联。
- 收藏与故事使用独立页面，不破坏时间轴的时间顺序。

### 6.4 P3：快捷记录与多媒体

- 桌面和锁屏小组件。
- 长按 App 图标进入“记录此刻”。
- 系统分享入口：从其他 App 导入文字、图片和网页链接。
- 系统快捷指令与语音助手入口。
- 语音记录。
- 语音转文字；可以同时保留原始音频和转写内容。
- 视频仅在真实使用需求明确后评估，不进入近期规划。

### 6.5 P4：账号与多设备

- 邮箱或系统账号登录。
- iOS 与 Android 多设备同步。
- 图片和附件同步。
- 离线变更队列、失败重试和同步状态展示。
- 编辑冲突副本和历史版本。
- 设备列表、远程退出和未同步数据提示。
- 账号注销、云端数据删除和完整数据导出。
- 端到端加密作为独立技术项目评估，不作未经实现的产品宣称。

### 6.6 AI 能力边界（远期评估）

- AI 不自动改写、覆盖或美化原始记录。
- 可以帮助用户用自然语言查找过去内容。
- 可以在用户主动触发时生成回顾，但必须与原始记录明确区分。
- 接入任何云端模型前，需要明确告知数据处理范围和用户授权。

### 明确不做

- 任何社交关系和公开可见范围
- 点赞、他人评论、转发和互动数据
- 内容热度排序
- 为提升使用时长而设计的无限推荐流

## 7. 数据模型建议

### 7.1 Entry（记录）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | UUID/String | 本地生成的唯一标识，为未来同步预留 |
| content | Text | 正文 |
| occurredAt | DateTime | 事情发生时间，用于时间轴排序 |
| createdAt | DateTime | 首次保存时间，不可覆盖 |
| updatedAt | DateTime | 最后编辑时间 |
| timezone | String | 创建或修改发生时间时的时区标识 |
| deletedAt | DateTime? | 回收站软删除时间，P1 启用 |

### 7.2 Attachment（附件，P1）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | UUID/String | 附件标识 |
| entryId | UUID/String | 所属记录 |
| type | Enum | image 等 |
| localPath | String | App 私有目录中的相对路径 |
| createdAt | DateTime | 添加时间 |
| sortOrder | Integer | 展示顺序 |

### 7.3 FollowUp（后续）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | UUID/String | 后续唯一标识，由客户端生成 |
| entryId | UUID/String | 所属记录 |
| content | Text | 后续正文 |
| createdAt | DateTime | 创建时间 |
| updatedAt | DateTime | 最后编辑时间 |
| deletedAt | DateTime? | 软删除/同步墓碑时间 |
| syncVersion | Integer/String | 同步版本，启用云同步时使用 |

### 7.4 Tag 与 EntryTag（P1）

标签与记录采用多对多关系，避免将标签拼成字符串保存在记录中。

### 7.5 同步元数据（P4）

Entry、FollowUp、Attachment 等需要同步的实体应预留：

- `id`：客户端 UUID。
- `updatedAt`：最近修改时间。
- `deletedAt`：软删除墓碑。
- `syncStatus`：pending、synced、failed 等本地状态。
- `serverVersion`：服务端版本或等效的并发控制标识。

不要把这些字段直接耦合到 UI；由同步层和 Repository 统一管理。

### 7.6 时间存储规则

- 数据库统一以 UTC 时间戳保存时间。
- 同时保留 IANA 时区标识，例如 `Asia/Shanghai`，用于准确还原当地时间。
- UI 按用户当前设置展示本地时间。
- `createdAt` 永远表示实际保存时刻；`occurredAt` 表示用户认为事情发生的时刻。

## 8. 非功能需求

### 8.1 性能

- 冷启动目标：主流近年设备上 2 秒内进入可操作状态。
- 保存文字记录后 300ms 内给予成功反馈。
- 时间轴采用分页或虚拟列表，不能一次渲染全部数据。
- 数据库为 `occurredAt`、`createdAt` 和搜索字段建立合适索引。

### 8.2 可靠性

- 写入操作采用数据库事务。
- 数据库结构升级必须提供迁移脚本，不可简单清库。
- 编辑过程中需要降低异常退出导致文本全部丢失的风险；P1 可加入本地草稿自动保存。
- 备份恢复完成后校验记录数量和附件完整性。

### 8.3 可访问性

- 支持系统字体缩放。
- 重要操作不能只依赖颜色表达。
- 点击区域符合手机触控尺寸要求。
- 深色模式下保证正文对比度。

### 8.4 隐私

- 默认不采集正文、图片、搜索词等内容数据。
- 若接入崩溃监控，必须过滤数据库内容和用户输入。
- 不申请与功能无关的权限。
- 相册、相机、通知和定位权限均在真正使用对应功能时再申请。

## 9. 技术选型

### 9.1 选型目标

当前阶段是个人练手项目和手机 App MVP，因此技术选择优先考虑：

1. 能较快完成一个真正可用的版本。
2. iOS 与 Android 尽量复用代码。
3. 离线数据库、图片、文件备份和生物识别生态成熟。
4. 调试、打包和后续维护成本可控。
5. 首版不被后端阻塞，但数据模型和分层必须支持未来多设备同步。
6. 具备走向正式上架产品所需的测试、迁移、监控和隐私基础。

### 9.2 默认推荐方案：React Native + Expo + TypeScript

如果你有 JavaScript、TypeScript、React 或前端基础，优先推荐此方案。

| 层级 | 建议 |
| --- | --- |
| App 框架 | React Native + Expo |
| 语言 | TypeScript，开启严格模式 |
| 路由 | Expo Router |
| 本地数据库 | SQLite |
| 数据访问 | 轻量 Repository 层；可选成熟 ORM，但不要让 ORM 主导领域模型 |
| 同步层 | 自建独立 Sync Service；UI 不直接访问云数据库 |
| 状态管理 | 首版使用 React Context/局部状态；复杂后再引入 Zustand 等轻量方案 |
| 表单 | 简单编辑页可直接使用受控状态，不必过早引入大型表单库 |
| 图片 | Expo 的图片选择、相机和文件系统能力 |
| 生物识别 | Expo Local Authentication 类能力 |
| 密钥保存 | 系统 Keychain/Keystore 对应的安全存储能力 |
| 测试 | 单元测试 + 核心流程组件测试；关键路径补端到端测试 |
| 后端 | 本地 MVP 不需要；同步版本推荐 Supabase 或等价的托管 Postgres/Auth/Storage 服务 |
| 服务端数据 | PostgreSQL + 对象存储，按用户隔离访问权限 |

#### 推荐原因

- 一套 TypeScript 代码可覆盖 iOS 和 Android。
- Expo 能减少原生工程配置，把时间放在产品功能上。
- SQLite 适合本地优先、结构化查询、按时间分页和后续数据迁移。
- 未来确实需要原生能力时，仍可以生成和维护原生项目，不必立即推翻重做。
- 本地数据库与同步服务分层后，既能在断网时完整使用，也能逐步增加多设备能力。

#### 注意事项

- “使用 Expo”不等于只能使用简单功能；但涉及数据库加密、复杂后台任务或特殊原生库时，需要验证其原生集成方式。
- 不要把日记正文放在 AsyncStorage 一类键值存储中；它适合偏好设置，不适合作为核心记录数据库。
- 不建议 MVP 同时引入 Redux、复杂 ORM、依赖注入框架和云端同步。
- Expo 适合起步和上线，并不意味着只是原型工具；但选择数据库加密、后台同步或特殊原生 SDK 前，要先做小型技术验证。

### 9.3 多设备与后端方案

#### 推荐演进方式

```text
页面/业务功能
      ↓
Repository（统一数据接口）
      ↓
本地 SQLite ← Sync Service → 云端 API/PostgreSQL/对象存储
```

- 页面只从本地数据库读取并写入，因此保存反馈快且始终可离线。
- Sync Service 监听本地待同步变更，与服务端进行增量双向同步。
- 云端不可直接成为页面的唯一数据源，否则弱网会直接影响写日记。
- Repository 和同步协议保持独立，未来可以更换云服务而不重写所有页面。

#### 为什么同步版推荐托管后端

对于个人开发者，账号、数据库、对象存储和权限控制都自己从零实现，工作量和安全风险较高。同步版本可以优先选择 Supabase 一类基于 PostgreSQL 的托管平台，快速获得：

- 用户认证与会话管理。
- PostgreSQL 结构化数据。
- 图片对象存储。
- 按用户隔离数据的服务端权限规则。
- 后续迁移到自建 PostgreSQL 的可能性。

这不是要求 MVP 立刻接入云端。建议先把本地版使用稳定，再实现同步协议。即使采用托管平台，仍需要自己认真设计权限、冲突处理、备份、限流、账户删除和隐私政策。

#### 不建议的方案

- 不建议把云端数据库 SDK 散落在页面组件中。
- 不建议以 AsyncStorage 作为记录数据库。
- 不建议把“每次启动时全量上传/下载”当作正式同步方案。
- 不建议在未解决密钥恢复前轻率宣称端到端加密。
- 不建议为了未来也许需要的功能，第一版就拆微服务。

### 9.4 其他方案何时更合适

#### Flutter + Dart

适合以下情况：

- 没有 React 技术包袱，愿意学习 Dart。
- 很重视跨平台 UI 一致性和自定义动画。
- 希望从移动端 UI 框架本身开始系统练习。

Flutter 同样能很好地完成本产品。它与 React Native 并不存在绝对优劣，主要差别是已有技能和你想练习的方向。若目标是尽快产出且已有前端经验，选 React Native；若想专门学习一套完整跨端 UI 体系，可以选 Flutter。

#### SwiftUI 或 Kotlin + Jetpack Compose

适合以下情况：

- 第一阶段只做一个平台。
- 目标是学习 iOS 或 Android 原生开发、系统能力和平台设计规范。
- 可以接受未来另一平台重新实现大部分界面与业务逻辑。

如果你只使用 iPhone，并且这个项目主要用于学习 iOS，那么 SwiftUI 反而可能是最合适的选择；同理，只做 Android 可选 Kotlin + Compose。

### 9.5 技术决策建议

可按下面的简单规则决定：

- 有 React/TypeScript 基础，并希望双端：选择 React Native + Expo。
- 没有明显技术偏好，但想学习跨平台移动 UI：选择 Flutter。
- 只做 iPhone 且想学 iOS：选择 SwiftUI。
- 只做 Android 且想学 Android：选择 Kotlin + Compose。

本 PRD 暂按以下方案实施：

- **客户端：React Native + Expo + TypeScript。** 同一代码库支持 iOS 和 Android。
- **本地数据：SQLite。** 本地优先，离线时完整可用。
- **架构：Repository + 独立 Sync Service。** 第一版实现 Repository，预留同步元数据；同步服务后续接入。
- **云端：同步版本再接入 Supabase 或等价的 PostgreSQL/Auth/Storage 服务。** MVP 不因后端延期。
- **Web：不作为首发端。** 未来需要时可复用服务端 API 和 TypeScript 领域模型，但不承诺与移动端复用全部 UI。

## 10. 建议工程结构

```text
src/
  app/                 # 页面与路由
  components/          # 通用 UI 组件
  features/
    entries/           # 新建、编辑、时间轴、详情
    followups/         # 添加后续与后续时间线
    calendar/          # 日历回看
    search/            # 搜索
    settings/          # 设置、备份、隐私
  database/
    migrations/        # 数据库迁移
    repositories/      # 数据访问封装
  services/            # 文件、备份、生物识别等系统能力
  sync/                # 同步队列、冲突处理与云端适配器（后续启用）
  theme/               # 颜色、字体和间距
  utils/               # 时间与通用工具
```

业务页面不应直接散落 SQL。统一通过 Repository 访问数据，既便于测试，也为未来同步或加密数据库留出替换空间。

## 13. 待确认的产品决策

以下问题不阻碍首版开工，但应在对应功能开发前确认：

1. 第一阶段同时支持 iOS 和 Android，还是先完成自己日常使用的平台？
2. 时间轴卡片默认展示全文，还是超过一定长度后折叠？
3. 删除是立即永久删除，还是首版就实现回收站？
4. 图片功能是否必须进入第一个可用版本？
5. 是否需要在 App 打开时默认启用生物识别锁？
6. 备份由用户手动保存到文件，还是后续接入系统云盘？
7. “拾时”的最终 Slogan 使用“把日子慢慢收好”，还是“把生活留在它发生的时刻”？
8. 首发同步是否支持邮箱登录、Apple/Google 登录，或两者同时提供？
9. 是否把端到端加密作为正式上线同步功能的硬性要求？这会直接影响密码找回和多设备密钥恢复设计。
10. “拾起一刻”首次上线只做随机回忆和那年今日，还是同时支持标签、地点筛选？
11. 关联记录和“故事”使用手动组织，还是允许系统提供关联建议？

## 14. 产品成功判断

个人项目不必用注册量或日活衡量。更合适的判断方式是：

- 自己能连续使用 30 天，并愿意继续记录。
- 大多数记录可以在产生想法后一分钟内完成。
- 能在一分钟内找到某一天或某个关键词对应的旧记录。
- 能通过“添加后续”持续补充一件事情的发展，同时保留原始记录当时的语境。
- “拾起一刻”能让旧记录被重新看见，而不会带来签到或任务压力。
- 更换手机或重装 App 前，能够可靠地备份并恢复全部数据。
- 开启同步后，可在 iOS/Android 多台设备之间可靠看到同一份记录，离线编辑也不会静默丢失。
- 使用过程中不会担心内容被意外公开或无法取回。
